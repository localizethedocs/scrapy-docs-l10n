蜘蛛中间件

The spider middleware is a framework of hooks into Scrapy's spider processing mechanism where you can plug custom functionality to process the responses that are sent to Spiders for processing and to process the requests and items that are generated from spiders.

激活旋转器中间件

To activate a spider middleware component, add it to the SPIDER_MIDDLEWARES setting, which is a dict whose keys are the middleware class path and their values are the middleware orders.

下面是一个示例：

天基ER_MIDDLEWARES = Power
    "myproject.middlewares.CustomSpiderMiddleware": 543,
}

The SPIDER_MIDDLEWARES setting is merged with the SPIDER_MIDDLEWARES_BASE setting defined in Scrapy (and not meant to be overridden) and then sorted by order to get the final sorted list of enabled middlewares: the first middleware is the one closer to the engine and the last is the one closer to the spider. In other words, the process_spider_input() method of each middleware will be invoked in increasing middleware order (100, 200, 300, ...), and the process_spider_output() method of each middleware will be invoked in decreasing order.

To decide which order to assign to your middleware see the SPIDER_MIDDLEWARES_BASE setting and pick a value according to where you want to insert the middleware. The order does matter because each middleware performs a different action and your middleware could depend on some previous (or subsequent) middleware being applied.

If you want to disable a builtin middleware (the ones defined in SPIDER_MIDDLEWARES_BASE, and enabled by default) you must define it in your project SPIDER_MIDDLEWARES setting and assign None as its value. For example, if you want to disable the off-site middleware:

天基ER_MIDDLEWARES = own
    "scrapy.spidermiddlewares.referer.Referer.Middlewares": None,
    "myproject.midglewares.CustomReferersSpiderMiddleware": 700,
}

最后，牢记某些中间件可能需要通过特定的设置启用。请参阅每个中间件文档以获取更多信息。

写入你自己的蜘蛛中间件

Each spider middleware is a component that defines one or more of these methods:

class scrapy.spidermiddlewares.SpiderMiddleware
async process_start(start: AsyncIterator[Any], /) → AsyncIterator[Any]

Iterate over the output of start() or that of the process_start() method of an earlier spider middleware, overriding it. For example:

async def process_start(自己, start):

        gow item_or_request

You may yield the same type of objects as start().

若要在低于2.13的Scrapy 版本上写入可迭代的中间器，请定义一个同步的 process_start_requests() 方法。例如：

def process_start_requests(self, start, spider):

process_spider_input(response)

这个方法需要通过旋转器中间件和旋转器进行处理。

process_spider_input() should return None or raise an exception.

如果返回 None ，废纸将继续处理这个响应，执行所有其他中间程序，直到最后将响应交给蜘蛛处理。

If it raises an exception, Scrapy won't bother calling any other spider middleware process_spider_input() and will call the request errback if there is one, otherwise it will start the process_spider_exception() chain. The output of the errback is chained back in the other direction for process_spider_output() to process it, or process_spider_exception() if it raised an exception.

参数:

response (Response object) -- 正在处理的响应

process_spider_output(response, result)

在处理了响应后，这个方法会与从Spider返回的结果一起调用。

process_spider_output() must return an iterable of Request objects and item objects.

Consider defining this method as an asynchronous generator, which will be a requirement in a future version of Scrapy. However, if you plan on sharing your spider middleware with other people, consider either enforcing Scrapy 2.7 as a minimum requirement of your spider middleware, or making your spider middleware universal so that it works with Scrapy versions earlier than Scrapy 2.7.

参数:

  • response (Response object) -- 从旋转器生成此输出的响应

  • result (an iterable of Request objects and item objects) -- 旋转器返回的结果

async process_spider_output_async(response, result)

If defined, this method must be an asynchronous generator, which will be called instead of process_spider_output() if result is an asynchronous iterable.

process_spider_exception(response, exception)

This method is called when a spider or process_spider_output() method (from a previous spider middleware) raises an exception.

process_spider_exception() should return either None or an iterable of Request or item objects.

If it returns None, Scrapy will continue processing this exception, executing any other process_spider_exception() in the following middleware components, until no middleware components are left and the exception reaches the engine (where it's logged and discarded).

If it returns an iterable the process_spider_output() pipeline kicks in, starting from the next spider middleware, and no other process_spider_exception() will be called.

参数:

  • response (Response object) -- 异常提起时正在处理的响应

  • exception (Exception object) -- 提出异常了

自定义蜘蛛中间体的基础类

Scrapy provides a base class for custom spider middlewares. It's not required to use it but it can help with simplifying middleware implementations and reducing the amount of boilerplate code in universal middlewares.

class scrapy.spidermiddlewares.base.BaseSpiderMiddleware(crawler: Crawler)[源代码]

可选的旋转中间层基类

Added in version 2.13.

这个类提供异步的 process_spider_output() 和 process_start() 方法的辅助方法。不需要使用这个类方法的中间路径。

You can override the get_processed_request() method to add processing code for requests and the get_processed_item() method to add processing code for items. These methods take a single request or item from the spider output iterable and return a request or item (the same or a new one), or None to remove this request or item from the processing.

get_processed_item(item: Any, response: Response | None) → Any[源代码]

从旋转器输出返回一个处理过的项目。

这个方法用单个项目来调用起始种子或旋转器输出。 它应该返回相同或不同的物品或 None 来忽略它。

参数:

  • item (item object) -- 输入项

  • response (Response object or None for start seeds) -- 正在处理的响应

返回:

处理过的项目或 None

get_processed_request(request: Request, response: Response | None) → Request | None[源代码]

从旋转器输出返回已处理的请求。

这个方法用单个请求从起始种子或旋转器输出调用。 它应该返回相同或不同的请求，或者 None 忽略它。

参数:

  • request (Request object) -- 输入请求

  • response (Response object or None for start seeds) -- 正在处理的响应

返回:

已处理的请求或 None

This page describes all spider middleware components that come with Scrapy. For information on how to use them and how to write your own spider middleware, see the spider middleware usage guide.

For a list of the components enabled by default (and their orders) see the SPIDER_MIDDLEWARES_BASE setting.

深中间件

class scrapy.spidermiddlewares.depth.DepthMiddleware[源代码]

DepthMiddleware 用于跟踪每个请求在被拆除的站点中的深度。它可以通过设置 request来操作。 eta['depth'] = 0 当没有之前设置的值 (通常只是第一个请求) 并以其他方式递增。

它可以用来限制最大的深度到擦除，根据其深度控制请求优先级以及类似的情况。

The DepthMiddleware can be configured through the following settings (see the settings documentation for more info):

  • DEPTH_LIMIT - The maximum depth that will be allowed to crawl for any site. If zero, no limit will be imposed.

  • DEPTH_STATS_VERBOSE - Whether to collect the number of requests for each depth.

  • DEPTH_PRIORITY - Whether to prioritize the requests based on their depth.

HttpErrorMiddleware

class scrapy.spidermiddlewares.httperror.HttpErrorMiddleware[源代码]

过滤不成功的HTTP响应(错误)，所以拼写器不必处理它们。 (大部分时间)造成间接费用，消耗更多资源，并使螺旋逻辑更加复杂。

根据`HTTP标准`_'，成功的答复是其身份码在200-300范围内的那些人。

If you still want to process response codes outside that range, you can specify which response codes the spider is able to handle using the handle_httpstatus_list spider attribute or HTTPERROR_ALLOWED_CODES setting.

例如，如果你想要你的蜘蛛来处理404个回复，你可以这样做：

从 scrapy.spiders 导入 CrawlSpider


类MySpider(CrawlSpider):
    handle_httpstatus_list = [404]

The handle_httpstatus_list key of Request.meta can also be used to specify which response codes to allow on a per-request basis. You can also set the meta key handle_httpstatus_all to True if you want to allow any response code for a request, and False to disable the effects of the handle_httpstatus_all key.

然而，要记住，处理非200个响应通常是一个坏主意，除非你真的知道你在做什么。

详情见：HTTP Status Code Definitions。

HttpErrorMiddleware 设置

HTTPERROR_ALLOWED_CODES

默认值： []

通过列表中包含非200条状态码的所有回复。

HTTPERROR_ALLOW_ALL

默认值： False

通过所有响应，无论其状态码如何。

class scrapy.spidermiddlewares.referer.ReferrerPolicy[源代码]

引用策略的抽象基类。

class scrapy.spidermiddlewares.referer.DefaultReferrerPolicy[源代码]

"no referrer-when en-downgrade"的变体，如果父请求使用 file:// 或 s3:// 方案，则"Referer"不会被发送。

警告

Scrapy 的默认引用策略——就像`no referrer-when en-downgrade`_'， 为浏览器推荐的 W3C 值 — — 将从任何 http(s):// 中发送一个非空的 "Referer" 头到任何 https:// URL，即使域是不同的。

`"相同的原始"_可能是一个更好的选择，如果您想要移除跨域请求的推荐信息。

class scrapy.spidermiddlewares.referer.NoReferrerPolicy[源代码]

https://www.w3.org/TR/referrer-policy/#referrer-policy-referrer

最简单的政策是“不要提到”， 其中规定不应向任何来源发送推荐人的信息和特定请求客户的请求。 标题将被完全省略。

class scrapy.spidermiddlewares.referer.NoReferrerWhenDowngradePolicy[源代码]

https://www.w3.org/TR/referrer-policy/#referrer-policy-no-referrer-when en-downgrade

"不推荐的时候降级"政策将一个完整的 URL 和来自一个 TLS 保护环境设置对象的请求发送到一个可能值得信任的 URL。 和来自不受TLS-保护的客户端的请求。

另一方面，由 TLS 保护的客户端向不可信任的 URL 的请求将不包含参考信息。Referer HTTP 头将不会发送。

这是用户代理的默认行为，如果没有指定其他策略。

备注

"不推荐当降级时"策略是 W3C 推荐的默认值，并被主要浏览器使用。

However, it is NOT Scrapy's default referrer policy (see DefaultReferrerPolicy).

class scrapy.spidermiddlewares.referer.SameOriginPolicy[源代码]

https://www.w3.org/TR/referrer-policy/#referrer-policy-some-original

“相同来源”策略规定了一个完整的 URL，用作推荐器， 当从特定请求客户端提出相同来源请求时，将作为推荐信息发送。

另一方面，跨源请求将不包含引用信息。Referer HTTP 标头将不会发送。

class scrapy.spidermiddlewares.referer.OriginPolicy[源代码]

https://www.w3.org/TR/referrer-policy/#referrer-policy-origin-policy-original

“原始”政策规定，在提出相同原始请求和来自特定请求客户的跨原始请求时，只将请求客户端的 ASCII 序列化作为参考信息发送。

class scrapy.spidermiddlewares.referer.StrictOriginPolicy[源代码]

https://www.w3.org/TR/referrer-policy/#referrer-policy-strict-original

“严格起源”政策在请求时发送请求客户端来源的ASCII序列化：- 从 TLS 受保护的环境设置对象到一个可能值得信任的 URL， - 从不受TLS-保护的环境设置对象到任何源.

另一方面，来自 TLS 保护的请求客户端对不可信的 URL 的请求将不包含参考信息。 Referer HTTP 头将不会发送。

class scrapy.spidermiddlewares.referer.OriginWhenCrossOriginPolicy[源代码]

https://www.w3.org/TR/referrer-policy/#referrer-policy-policy-origin-when-cross-origin-

“原始跨源”政策规定，完整的 URL，可作为推荐人使用， 当从特定请求客户端提出相同来源请求时，作为推荐信息发送， 并且只有ASCII序列化请求客户端的来源，在从特定请求客户端提出交叉请求时作为参考信息发送。

class scrapy.spidermiddlewares.referer.StrictOriginWhenCrossOriginPolicy[源代码]

https://www.w3.org/TR/referrer-policy/#referrer-policy-strict-origin-when-cross-origin-

“严格源头跨源”政策规定，完整的 URL，可作为推荐人使用， 当从特定请求客户端提出相同来源请求时，作为推荐信息发送， 并且只有ASCII在提出跨源请求时将请求客户端的来源序列化：

  • 从 TLS 受保护的环境设置对象到一个可能值得信任的 URL，和

  • 从非 TLS 保护的环境设置对象到任何来源。

另一方面，受TLS保护的客户端对不可信的 URL 的请求将不包含参考信息。 Referer HTTP 头将不会发送。

class scrapy.spidermiddlewares.referer.UnsafeUrlPolicy[源代码]

https://www.w3.org/TR/referrer-policy/#referrer-policy-unsafe-url

“不安全-url”策略规定，一个完整的 URL，可作为推荐人使用。 与不同来源的请求和来自特定请求客户的相同请求一起发送。

注意：该策略的名称不会是假的；它是不安全的。该策略将泄露来源和路径，从受TLS-保护的资源到不安全的来源。 认真考虑为潜在敏感文件制定这样一项政策所产生的影响。

警告

不推荐“不安全-url”政策。
